# Design — RF-06.1: Calendário Visual do Aluno

**Data:** 2026-07-13
**Status:** Aprovado

---

## Contexto

O PRD exige visualização em calendário para o aluno (RF-06.1 dia/semana/mês, RF-06.2 cores por status, RF-06.3 filtros). O aluno hoje tem:
- `ScheduleView` — lista de horários disponíveis para reservar
- `BookingsView` — tabela das próprias reservas

O backend já retorna `startsAt`, `endsAt`, `type`, `groupClassName`, `trainerName` no endpoint `GET /api/bookings/my` via `BookingResponse`, mas o tipo frontend `Booking` não expõe esses campos.

---

## Escopo

- Adicionar campos faltantes ao tipo `Booking` em `api.ts`
- Criar `CalendarView.vue` no namespace do aluno (view semanal, read-only)
- Adicionar rota `/student/calendar` e item no menu do aluno
- **Não** implementar modos dia/mês (YAGNI — o PRD lista como requisito mas o trainer também só tem semana; consistência é mais importante que cobertura total de modos)
- **Sem filtros** — o aluno vê apenas suas próprias reservas; filtro por trainer/tipo aumentaria escopo sem ganho real

---

## Design

### 1. Tipo `Booking` atualizado

Em `frontend/src/types/api.ts`, adicionar os campos retornados pelo backend mas ausentes no tipo:

```ts
export interface Booking {
  id: string
  scheduleId: string
  studentId: string
  status: BookingStatus
  bookedAt: string
  // campos do Schedule enriquecidos pelo backend
  startsAt: string
  endsAt: string
  type: ScheduleType
  groupClassName: string | null
  trainerName: string | null
}
```

### 2. Nova view `CalendarView.vue`

Arquivo: `frontend/src/views/student/CalendarView.vue`

**Estrutura de navegação semanal** — idêntica ao trainer:
- Botões `← | label da semana | → | Hoje`
- `weekStart` começa na segunda-feira da semana atual
- `loadWeek()` chama `bookingStore.fetchMyBookings()` (sem parâmetros — retorna todas as reservas do aluno; filtragem da semana é feita no frontend)

**Grade semanal** — 7 colunas (Seg–Dom), mesmo CSS pattern do trainer:
- `day-col`, `day-header`, `day-header.today`, `slots`, `slot-card`
- `slotsForDay(iso)` filtra `bookingStore.bookings` por `b.startsAt.startsWith(iso)`
- Ordenado por `startsAt`

**Cards por status** (RF-06.2):
- `CONFIRMED` → classe `status-confirmed` → borda esquerda `#10b981` (verde)
- `PENDING` → classe `status-pending` → borda esquerda `#f59e0b` (amarelo)
- `CANCELLED` → classe `status-cancelled` → `opacity: 0.5`, borda esquerda `#9ca3af` (cinza)

Conteúdo do card:
```
08:00 – 09:00
Aula coletiva / Personal
[nome da turma ou nome do trainer]
```

**Sem ações** — calendário read-only. Para cancelar, o aluno usa `BookingsView`.

**Estado vazio por dia:** `—` centralizado (igual ao trainer).

### 3. Rota

Em `frontend/src/router/index.ts`, adicionar dentro do bloco de rotas protegidas do aluno:

```ts
{
  path: '/student/calendar',
  component: () => import('@/views/student/CalendarView.vue'),
  meta: { requiresAuth: true, role: 'STUDENT' },
},
```

### 4. Menu (AppLayout)

Inserir entre o link "Minhas Reservas" (`/student/bookings`) e o link "Aulas Disponíveis" (`/student/schedule`):

```html
<RouterLink to="/student/calendar" class="nav-item" active-class="active"
  title="Meu Calendário" aria-label="Meu Calendário" @click="mobileOpen = false">
  <i class="pi pi-calendar-plus" />
</RouterLink>
```

Ícone `pi-calendar-plus` (diferencia do `pi-calendar` já usado em "Aulas Disponíveis").

## Critérios de aceite

- Nova rota `/student/calendar` acessível pelo aluno logado
- Ícone "Meu Calendário" aparece no menu lateral do aluno
- Grade semanal com 7 colunas (Seg–Dom), navegação prev/next/hoje
- Dia atual destacado (fundo azul no cabeçalho)
- Reservas do aluno aparecem nos dias corretos
- Cards com cor por status: verde (CONFIRMED), amarelo (PENDING), cinza + opaco (CANCELLED)
- Cada card mostra horário, tipo e nome da turma/trainer
- Dias sem reserva mostram `—`
- View é read-only (sem botões de ação nos cards)
